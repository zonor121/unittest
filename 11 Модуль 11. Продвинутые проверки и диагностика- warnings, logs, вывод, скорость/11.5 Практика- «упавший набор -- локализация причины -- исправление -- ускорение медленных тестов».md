# От красного набора к ясной причине: как в `unittest` пройти путь «падение → локализация → исправление → ускорение»

Упавший набор редко ломается только в одном месте. Обычно проблема тройная: один тест падает, вывод зашумлён лишними `print()`, а несколько кейсов ещё и тянут прогон вниз по времени. В стандартном `unittest` для этого уже есть рабочий минимум: `-b/--buffer` прячет вывод успешных тестов и прикладывает вывод неуспешных к тексту ошибки, `--locals` добавляет локальные переменные в traceback, а `--durations N` показывает самые медленные тесты. В актуальной документации Python это штатные CLI-опции `unittest`; `--locals` существует с Python 3.5, `--durations` — с Python 3.12. В Python 3.14 вывод `unittest` ещё и раскрашивается по умолчанию, поэтому примеры ниже я показываю в упрощённом монохромном виде. ([Python documentation][1])

## Введение

В этой практике мы не будем брать абстрактные куски кода. Вместо этого построим один маленький проект, который специально устроен неудобно: в нём есть реальный баг в прод-коде, один неудачно написанный тест, warning как часть совместимости, логирование как часть диагностики и пара тестов, которые тормозят из-за настоящего ожидания времени. Дальше мы пройдём весь цикл целиком: сначала увидим плохой красный набор, потом улучшим сам сигнал падения через `subTest()` и осмысленные `assert`-сообщения, затем исправим код и в конце ускорим тесты, опираясь на `--durations`. `python -m unittest` без аргументов эквивалентен `python -m unittest discover`, а если Вам нужны параметры discovery вроде `-s`, `-p` и `-t`, субкоманду `discover` нужно вызывать явно. ([Python documentation][1])

Смысл этой темы в том, что хороший тестовый набор диагностирует проблему в два шага, а не в шесть. Он не заставляет Вас каждый раз запускать отладчик только для того, чтобы узнать входные данные или понять, какой элемент таблицы сломался. Для этого в `unittest` уже есть `subTest()`, стандартные diff в `assertEqual()` для встроенных типов, пользовательские `msg`, настройка `longMessage` и `maxDiff`. `subTest()` показывает параметры подслучая при падении, успешные подслучаи не шумят, а неуспешные по умолчанию записываются как обычные failures. `longMessage=True` используется по умолчанию, поэтому Ваш `msg` дописывается к стандартному тексту ошибки, а `maxDiff` управляет длиной diff и может быть снят установкой `None`. ([Python documentation][1])

```mermaid
flowchart LR
    A["Запуск набора"] --> B["-b / --buffer"]
    B --> C["--locals"]
    C --> D["Переписать слабый тест:\nsubTest + assertEqual + msg"]
    D --> E["Исправить прод-код"]
    E --> F["--durations=0 -v"]
    F --> G["Убрать реальные ожидания\nи ускорить тесты"]
```

## Мини-проект, который будет падать

Ниже — модуль, с которым мы будем работать. Он небольшой, но в нём есть всё нужное для практики: нормализация имени пользователя, предупреждение о deprecated-сценарии, информационный лог, ошибка в преобразовании роли и функция ожидания готовности задачи, которая в тестах может стать медленной, если использовать реальное время.

```python
# app/user_profile.py
import logging
import time
import warnings

logger = logging.getLogger(__name__)


def normalize_username(raw: str) -> str:
    clean = raw.strip().lower()
    if len(clean) < 3:
        warnings.warn(
            "usernames shorter than 3 characters are deprecated and will be rejected in v2",
            DeprecationWarning,
            stacklevel=2,
        )
    return clean


def build_profile(payload: dict) -> dict:
    logger.info("building profile for user_id=%s", payload["id"])
    username = normalize_username(payload["username"])
    role = payload.get("role", "user").upper()  # BUG: здесь ошибка
    return {
        "id": payload["id"],
        "username": username,
        "role": role,
    }


def wait_until_ready(
    check_status, *, timeout: float = 0.30, interval: float = 0.10
) -> bool:
    deadline = time.monotonic() + timeout

    while time.monotonic() < deadline:
        if check_status() == "ready":
            logger.info("job became ready")
            return True
        time.sleep(interval)

    logger.error("job did not become ready before timeout")
    return False
```

На уровне контракта здесь уже можно выделить четыре независимых сигнала. `normalize_username()` должна выдавать `DeprecationWarning` на слишком короткое имя; `build_profile()` должна писать информационный лог; `wait_until_ready()` на таймауте должна писать error-лог; а медленные тесты вокруг `wait_until_ready()` будут хорошо видны через `--durations`, потому что `unittest` хранит elapsed time каждого теста в `collectedDurations`, а `addDuration()` считает время теста вместе с cleanup-функциями. Для логов в `unittest` есть `assertLogs()` и `assertNoLogs()`: первый проверяет, что внутри блока был хотя бы один лог на указанном logger’е или его дочерних logger’ах с заданным минимальным уровнем и возвращает helper с `records` и `output`; второй проверяет, что таких логов не было, и ничего не возвращает. Для предупреждений есть `assertWarns()` и `assertWarnsRegex()`, которые работают независимо от текущих warning-фильтров. ([Python documentation][1])

## Плохой исходный набор: код сломан, а тест ещё и мешает

Теперь посмотрим на намеренно плохую первую версию тестов. Она вполне типична для реальных проектов: в цикле идёт `assertTrue(...)`, в тесте есть отладочные `print()`, а ожидание готовности задачи проверяется через реальные `sleep()`.

```python
# tests/test_user_profile_initial.py
import unittest

from app.user_profile import build_profile, wait_until_ready


class TestNoise(unittest.TestCase):
    def test_green_but_chatty(self):
        print("debug: connected to fake service")
        print("debug: ready")
        self.assertTrue(True)


class TestBuildProfileInitial(unittest.TestCase):
    def test_examples(self):
        cases = [
            {
                "payload": {"id": 1, "username": " Alice ", "role": "ADMIN"},
                "expected": {"id": 1, "username": "alice", "role": "admin"},
            },
            {
                "payload": {"id": 2, "username": " Bob "},
                "expected": {"id": 2, "username": "bob", "role": "user"},
            },
        ]

        for case in cases:
            print("checking payload", case["payload"])
            self.assertTrue(build_profile(case["payload"]) == case["expected"])


class TestWaitUntilReadySlow(unittest.TestCase):
    def test_ready_after_two_polls(self):
        states = iter(["pending", "pending", "ready"])

        def check():
            return next(states, "ready")

        self.assertTrue(wait_until_ready(check, timeout=0.30, interval=0.10))

    def test_timeout(self):
        self.assertFalse(
            wait_until_ready(lambda: "pending", timeout=0.25, interval=0.05)
        )
```

Проблема здесь двойная. Во-первых, прод-код действительно содержит дефект: `role` переводится в верхний регистр, а ожидается нижний. Во-вторых, сам тест написан так, что он плохо объясняет падение. `assertTrue(build_profile(...) == expected)` не использует типоспецифическое сравнение `unittest`, а значит теряет автоматический diff для словарей. В результате failure будет намного беднее, чем с обычным `assertEqual(actual, expected)`, который умеет подключать детализированное сравнение для встроенных типов. А поскольку внутри цикла нет `subTest()`, из сообщения об ошибке также не видно, какой именно кейс из таблицы сломан. ([Python documentation][1])

## Первый прогон: сначала уберите шум

Если запустить такой набор напрямую, Вы, скорее всего, увидите смесь полезной и бесполезной консольной информации. Для первого прохода лучше уже на старте дать `-b` и `--locals`:

```bash
python -m unittest discover -b --locals -v
```

`-b/--buffer` буферизует `stdout` и `stderr`: вывод успешных тестов отбрасывается, а вывод неуспешных нормальным образом прикладывается к тексту failure или error. `--locals` добавляет локальные переменные в traceback. На уровне API этим же управляют `buffer=True` и `tb_locals=True` у `TextTestRunner`, а в `TestResult` есть одноимённые атрибуты `buffer` и `tb_locals`. ([Python documentation][1])

На уровне реализации это не просто “другая печать”. В CPython при `buffer=True` `TestResult` между `startTest()` и `stopTest()` временно подменяет `sys.stdout` и `sys.stderr` на `io.StringIO()`, а если тест падает, накопленный вывод дописывается в текст ошибки и может быть зеркалирован обратно в реальные потоки как блоки `Stdout:` и `Stderr:`. Для `--locals` traceback строится через `traceback.TracebackException(..., capture_locals=self.tb_locals, compact=True)`, то есть локальные переменные включаются в форматируемый traceback на уровне самого результата теста. ([GitHub][2])

После такого запуска шум от `test_green_but_chatty` исчезнет, потому что успешный тест больше не печатает свои `print()` в живую консоль. Это уже хорошо. Но Ваша главная failure всё ещё будет слишком слабой. Она будет в духе `False is not true`, и даже с `--locals` Вы увидите только то, что в каком-то месте цикла булево сравнение не сошлось. То есть `-b` и `--locals` подняли качество сигнала, но сам тест всё ещё сформулирован неудачно. Здесь важная педагогическая мысль: иногда сначала нужно чинить не production-код, а сам тест, чтобы он начал объяснять дефект. ([Python documentation][1])

## Сначала исправьте тест, а уже потом код

Ниже — вторая версия того же теста. Она делает ровно ту же логическую проверку, но строит качественно другой failure. Здесь есть `subTest()` с именем кейса и исходным `payload`, `assertEqual()` вместо `assertTrue()`, доменное `msg=` и `maxDiff = None`, чтобы не обрезать diff, если словари станут больше.

```python
# tests/test_user_profile.py
import unittest

from app.user_profile import build_profile


class TestBuildProfile(unittest.TestCase):
    maxDiff = None

    def test_examples(self):
        cases = [
            {
                "case": "admin role should be lower-cased",
                "payload": {"id": 1, "username": " Alice ", "role": "ADMIN"},
                "expected": {"id": 1, "username": "alice", "role": "admin"},
            },
            {
                "case": "default role should stay user",
                "payload": {"id": 2, "username": " Bob "},
                "expected": {"id": 2, "username": "bob", "role": "user"},
            },
        ]

        for case in cases:
            with self.subTest(case=case["case"], payload=case["payload"]):
                self.assertEqual(
                    build_profile(case["payload"]),
                    case["expected"],
                    msg="build_profile() должен нормализовать username и приводить role к нижнему регистру",
                )
```

Здесь срабатывают сразу несколько механизмов `unittest`. `subTest(msg=None, **params)` показывает `msg` и переданные параметры, если подслучай падает; успешные подслучаи по умолчанию не делают ничего, а неуспешные записываются как обычные failures. `assertEqual()` для значений одного и того же встроенного типа подключает типоспецифическое сравнение, поэтому для словаря Вы получите не абстрактное `False is not true`, а diff словарей. Поскольку `longMessage=True` — значение по умолчанию — Ваш `msg` допишется к стандартному сообщению об ошибке, а не заменит его. `maxDiff = None` снимает лимит длины diff, который по умолчанию равен `80*8` символам для методов, печатающих различия, включая `assertDictEqual()` и `assertMultiLineEqual()`. ([Python documentation][1])

Теперь failure превращается из “что-то не так” в очень конкретный диагноз. Упрощённо он будет выглядеть так:

```text
FAIL: test_examples (...) (case='admin role should be lower-cased',
payload={'id': 1, 'username': ' Alice ', 'role': 'ADMIN'})

AssertionError:
{'id': 1, 'username': 'alice', 'role': 'ADMIN'}
!=
{'id': 1, 'username': 'alice', 'role': 'admin'}
: build_profile() должен нормализовать username и приводить role к нижнему регистру
```

Из такого сообщения уже видно всё, что нужно для исправления: какой именно кейс упал, какие поля в результате отличаются и какое бизнес-правило нарушено. Никакого дополнительного `print()`, никакого дебаггера и никакого гадания “а какой элемент таблицы сейчас был в цикле”. Это и есть правильное применение `subTest()` и `msg=`: не раздувать вывод, а сделать failure самодостаточным. ([Python documentation][1])

## Теперь исправьте код

После того как падение стало говорящим, сам дефект локализуется почти мгновенно. Ошибка в `build_profile()` — это строка с `upper()`. Исправление выглядит так:

```python
# app/user_profile.py
def build_profile(payload: dict) -> dict:
    logger.info("building profile for user_id=%s", payload["id"])
    username = normalize_username(payload["username"])
    role = payload.get("role", "user").lower()
    return {
        "id": payload["id"],
        "username": username,
        "role": role,
    }
```

На этом этапе полезно сделать ещё один шаг, который часто пропускают. Если warning и лог действительно являются частью контракта, их нужно фиксировать отдельными тестами, а не надеяться на ручной просмотр консоли. Для предупреждений `unittest` даёт `assertWarns()` и `assertWarnsRegex()`; они могут работать как context manager, а `assertWarns()` дополнительно сохраняет warning-объект, `filename` и `lineno`. Для логирования есть `assertLogs()` и `assertNoLogs()`: `assertLogs()` проверяет, что внутри блока был хотя бы один лог на указанном logger’е или его children с уровнем не ниже заданного и возвращает helper с `records` и `output`; `assertNoLogs()` проверяет, что таких логов не было, и ничего не возвращает. ([Python documentation][1])

Ниже — компактная финальная версия таких тестов:

```python
# tests/test_user_profile_contracts.py
import unittest
from unittest.mock import patch

from app.user_profile import (
    build_profile,
    normalize_username,
    wait_until_ready,
)


class TestWarningsAndLogs(unittest.TestCase):
    def test_warns_on_short_username(self):
        with self.assertWarnsRegex(
            DeprecationWarning,
            "shorter than 3 characters",
        ):
            normalize_username(" Bo ")

    def test_logs_profile_build(self):
        with self.assertLogs("app.user_profile", level="INFO") as cm:
            build_profile({"id": 7, "username": " Alice ", "role": "ADMIN"})

        self.assertEqual(
            cm.records[0].getMessage(),
            "building profile for user_id=7",
        )

    def test_ready_path_has_no_error_logs(self):
        with (
            patch("app.user_profile.time.sleep"),
            patch("app.user_profile.time.monotonic", side_effect=[0.00, 0.00]),
        ):
            with self.assertNoLogs("app.user_profile", level="ERROR"):
                result = wait_until_ready(
                    lambda: "ready",
                    timeout=0.30,
                    interval=0.10,
                )

        self.assertTrue(result)
```

Эти тесты лучше простого чтения консоли по двум причинам. Во-первых, они превращают warning и logging из побочного эффекта в контракт. Во-вторых, они не зависят от случайной конфигурации терминала и не требуют смотреть на живой stream во время запуска. `assertWarns()` работает независимо от warning-фильтров, а `assertLogs()` проверяет точный logger, уровень и, при необходимости, сами `LogRecord` через `cm.records`. Это куда надёжнее, чем надеяться, что кто-то заметит одинокую строку в огромном CI-логе. ([Python documentation][1])

## Ускорение: сначала измерьте, потом оптимизируйте

После исправления функционального бага набор может стать зелёным, но всё ещё медленным. На этом месте многие начинают наугад переписывать тесты. В `unittest` лучше идти в обратном порядке: сначала измерить, потом трогать код. Для этого и существует `--durations N`. Опция показывает `N` самых медленных тестов; если передать `0`, будет показан весь список. В `unittest` эта функция появилась в Python 3.12, а сами durations хранятся в `TestResult.collectedDurations`. На стороне `TextTestRunner` список сортируется по убыванию времени, а если verbosity ниже 2, durations меньше `0.001` секунды скрываются и раннер пишет подсказку “use -v to show these durations”. ([Python documentation][1])

Для этой практики команда выглядит так:

```bash
python -m unittest discover --durations=0 -v
```

Если прогнать наш набор в исходной медленной версии, в конце Вы увидите, что в хвосте почти наверняка окажутся оба теста на `wait_until_ready()`. Это ожидаемо: они реально ждут через `time.sleep()`. И здесь очень важно помнить одну деталь из документации `unittest`: `addDuration(test, elapsed)` получает время теста **вместе** с cleanup-функциями. То есть `--durations` показывает не только “медленное тело теста”, но и медленную уборку ресурсов после него. В нашем примере cleanup нет, но это правило полезно знать заранее, потому что на реальных наборах хвост часто живёт именно в teardown или cleanup. ([Python documentation][1])

Вот медленная версия тестов, которую `--durations` как раз и подсветит:

```python
# tests/test_user_profile_slow.py
import unittest

from app.user_profile import wait_until_ready


class TestWaitUntilReadySlow(unittest.TestCase):
    def test_ready_after_two_polls(self):
        states = iter(["pending", "pending", "ready"])

        def check():
            return next(states, "ready")

        self.assertTrue(wait_until_ready(check, timeout=0.30, interval=0.10))

    def test_timeout(self):
        self.assertFalse(
            wait_until_ready(lambda: "pending", timeout=0.25, interval=0.05)
        )
```

Такие тесты легко написать и очень легко пропустить в ревью, потому что каждая отдельная `sleep()` кажется короткой. Но в сумме они начинают заметно портить прогон. Правильное исправление здесь не в том, чтобы “сделать timeout чуть меньше”. Правильное исправление — убрать реальное ожидание из unit-теста. Поскольку предыдущий модуль курса уже разбирал мокирование времени, здесь логично использовать ту же технику: подменить `time.sleep()` и `time.monotonic()` в модуле под тестом. Тогда бизнес-логика цикла останется реальной, а wall-clock ожидание исчезнет. Само решение про патч времени — это уже инженерный вывод из предыдущих тем; `--durations` здесь нужен, чтобы сначала доказать, что проблема вообще есть и где именно она живёт. ([Python documentation][1])

Ниже — быстрая версия тех же тестов:

```python
# tests/test_user_profile_fast.py
import unittest
from unittest.mock import patch

from app.user_profile import wait_until_ready


class TestWaitUntilReadyFast(unittest.TestCase):
    def test_ready_after_two_polls(self):
        states = iter(["pending", "pending", "ready"])

        def check():
            return next(states)

        with (
            patch("app.user_profile.time.sleep") as mocked_sleep,
            patch(
                "app.user_profile.time.monotonic", side_effect=[0.00, 0.00, 0.10, 0.20]
            ),
        ):
            with self.assertLogs("app.user_profile", level="INFO") as cm:
                result = wait_until_ready(
                    check,
                    timeout=0.30,
                    interval=0.10,
                )

        self.assertTrue(result)
        self.assertEqual(mocked_sleep.call_count, 2)
        self.assertEqual(cm.records[-1].getMessage(), "job became ready")

    def test_timeout(self):
        with (
            patch("app.user_profile.time.sleep") as mocked_sleep,
            patch(
                "app.user_profile.time.monotonic",
                side_effect=[0.00, 0.00, 0.10, 0.20, 0.31],
            ),
        ):
            with self.assertLogs("app.user_profile", level="ERROR") as cm:
                result = wait_until_ready(
                    lambda: "pending",
                    timeout=0.30,
                    interval=0.10,
                )

        self.assertFalse(result)
        self.assertEqual(mocked_sleep.call_count, 3)
        self.assertEqual(cm.records[0].levelname, "ERROR")
        self.assertEqual(
            cm.records[0].getMessage(),
            "job did not become ready before timeout",
        )
```

Эти тесты делают несколько правильных вещей одновременно. Они больше не ждут реальное время, но всё ещё проверяют логику цикла: сколько было “снов”, где наступил таймаут, на каком уровне был написан лог и какой именно текст получил `LogRecord`. После такого рефакторинга `--durations=0 -v` покажет, что хвост почти исчез. А если в наборе останутся ещё какие-то медленные тесты, Вы увидите их поимённо и сможете разбирать следующий хвост так же последовательно, а не наугад. По исходникам `TextTestRunner` durations печатаются только после выполнения набора, сортируются по убыванию и, при verbosity `< 2`, скрывают записи короче `0.001` секунды — именно поэтому в практическом режиме полезно сочетать `--durations=0` с `-v`. ([GitHub][3])

## Что важно вынести из этого сценария

В этой практике есть одна центральная мысль. Диагностика почти всегда дешевле переписывания кода вслепую. `-b` сразу отделяет шум успешных тестов от вывода действительно проблемного кейса. `--locals` показывает состояние кадра в момент падения. `subTest()` и нормальные assert-методы превращают абстрактное “что-то в цикле не сошлось” в адресное сообщение. `assertWarnsRegex()` и `assertLogs()` фиксируют контракт warning- и log-сигналов отдельно от консоли. `--durations` поднимает наверх хвосты времени, включая cleanup. И только после этого имеет смысл исправлять код и ускорять тесты. Все эти шаги встроены в стандартный `unittest`; никакой внешний фреймворк для такого рабочего цикла не обязателен. ([Python documentation][1])

## Заключение

Практика модуля 11.5 не про ещё один assert и не про ещё один CLI-флаг. Она про дисциплину работы с падающим набором. Сначала Вы улучшаете сигнал: убираете шум через `-b`, раскрываете состояние через `--locals`, делаете failure говорящим через `subTest()`, `assertEqual()` и короткое доменное `msg`. Затем фиксируете контракт вокруг warnings и логов. И только потом переходите к скорости: измеряете хвосты через `--durations`, а не оптимизируете вслепую. Такой порядок почти всегда быстрее, чем прямой прыжок в код. ([Python documentation][1])

Хороший тестовый набор не просто ловит баги. Он помогает чинить их с минимальным числом лишних запусков. Когда падение само показывает подслучай, diff, доменный смысл, warning, лог и имя самого медленного теста, диагностика перестаёт быть отдельной фазой проекта. Она становится естественным свойством набора. Именно это и стоит считать итогом темы 11.5. ([Python documentation][1])

## Практическое задание

### Цель

Написать мини-проект и набор тестов, который проходит полный цикл из этого лонгрида: сначала падает неудобно, затем после улучшения тестов даёт точную локализацию причины, после исправления прод-кода становится зелёным и, наконец, ускоряется за счёт удаления реального ожидания времени. При решении используйте только стандартную библиотеку Python и `unittest`.

### Задание (шаги)

1. Создайте модуль `app/user_profile.py` со следующими функциями: `normalize_username(raw: str)`, `build_profile(payload: dict)` и `wait_until_ready(check_status, *, timeout, interval)`.
   В `normalize_username()` обрежьте пробелы и приведите имя к нижнему регистру. Если длина результата меньше 3, выдайте `DeprecationWarning`.
   В `build_profile()` залогируйте `INFO`-сообщение, нормализуйте имя пользователя и верните словарь профиля. Намеренно заложите баг: приводите `role` не к тому регистру, который ожидают тесты.
   В `wait_until_ready()` используйте `time.monotonic()` и `time.sleep()`; на успехе пишите `INFO`-лог, на таймауте — `ERROR`.

2. Напишите первую, намеренно слабую версию тестов:
   - один тест с циклом по кейсам через `assertTrue(actual == expected)`;
   - хотя бы один `print()` внутри passing-теста;
   - два теста на `wait_until_ready()`, которые используют реальные задержки.

3. Запустите набор через:

   и посмотрите, что именно стало лучше, а что всё ещё плохо читается.

4. Перепишите слабый табличный тест так, чтобы он использовал `subTest()`, `assertEqual()` и короткое доменное `msg=`. При необходимости поставьте `maxDiff = None`.

5. После того как failure станет точным, исправьте production-баг в `build_profile()`.

6. Добавьте отдельные тесты на warning и логи:
   - `assertWarnsRegex()` для короткого имени пользователя;
   - `assertLogs()` для `build_profile()` и для таймаута в `wait_until_ready()`;
   - `assertNoLogs()` для happy path в `wait_until_ready()` на уровне `ERROR`.

7. Измерьте скорость набора:

   ```bash
   python -m unittest discover --durations=0 -v
   ```

   Найдите хвосты времени.

8. Перепишите медленные тесты на `wait_until_ready()` так, чтобы они не ждали реальное время. Используйте `patch()` для `time.sleep()` и `time.monotonic()` в модуле под тестом.

9. Повторно запустите набор с `--durations=0 -v` и убедитесь, что slow-tail сократился.

### Подсказки по ключевым частям

Для подслучаев используйте не просто имя кейса, а и его полезные параметры. В `subTest()` можно передать `case=...`, `payload=...` и другие значения; они будут показаны в тексте failure, если конкретный подслучай упадёт, а успешные подслучаи не создадут дополнительного шума. По документации `unittest` именно так и задуман `subTest()`, а его failures по умолчанию записываются как обычные failures. ([Python documentation][1])

Не дублируйте diff в `msg=`. Оставьте встроенному `assertEqual()` обязанность показать, что именно не совпало, а в `msg=` напишите одно короткое бизнес-правило. Это хорошо работает потому, что `longMessage=True` используется по умолчанию, и `msg` будет добавлен к стандартному сообщению об ошибке, а не заменит его. Если diff у словаря или многострочной строки режется, используйте `maxDiff = None`. ([Python documentation][1])

Для warning-теста берите `assertWarnsRegex()`, а не ручной просмотр консоли. Этот assertion работает независимо от warning-фильтров, поэтому тест не зависит от того, как именно настроен запуск Python в Вашей среде. ([Python documentation][1])

Для логов работайте через конкретный logger модуля, а не через root logger. `assertLogs()` проверяет сообщения на указанном logger’е и его children, с минимальным уровнем по умолчанию `INFO`, и возвращает helper с `records` и `output`. Это позволяет проверять не только строку, но и уровень, имя logger’а и итоговое сообщение через `record.getMessage()`. `assertNoLogs()` полезен для “тихих” happy-path веток. ([Python documentation][1])

Для скорости не оптимизируйте вслепую. `--durations N` показывает самые медленные тесты, а `--durations 0` — все. По исходникам `TextTestRunner`, в невverbose-режиме durations короче `0.001` секунды скрываются, поэтому для полного списка удобно добавлять `-v`. Помните и о том, что длительность теста включает cleanup-функции. ([Python documentation][1])

### Что проверить перед отправкой (чек-лист)

- В наборе есть хотя бы одна исходная failure, которую потом можно локализовать и исправить.
- Для локализации Вы реально использовали `-b` и `--locals`, а не только переписали код. По документации `-b` должен скрывать вывод passing-тестов и прикладывать вывод failing-тестов к failure/error, а `--locals` — показывать локальные переменные в traceback. ([Python documentation][1])
- Табличный тест переписан через `subTest()`, а не через немой цикл с `assertTrue(...)`. ([Python documentation][1])
- В улучшенной версии тестов используется `assertEqual()` с понятным `msg=`, а не сравнение через `assertTrue(actual == expected)`. `unittest` умеет строить более полезные сообщения для встроенных типов именно через корректные assert-методы. ([Python documentation][1])
- Warning проверяется через `assertWarnsRegex()`, а логирование — через `assertLogs()` / `assertNoLogs()`. ([Python documentation][1])
- Замер скорости сделан через `--durations`, а медленные тесты действительно переписаны так, чтобы убрать реальное ожидание времени. ([Python documentation][1])
- После рефакторинга набор остаётся независимым: тесты не зависят друг от друга, от текущего времени и от последовательности запуска.

### Советы по улучшению работы

Сделайте два коммита, а не один. В первом коммите улучшите только сами тесты и диагностику: `subTest()`, `assertEqual()`, `msg=`, `assertWarnsRegex()`, `assertLogs()`, `-b`, `--locals`. Во втором исправьте production-баг и ускорьте slow-тесты. Такой порядок делает историю изменений понятной и показывает, что сильный тест действительно локализовал проблему до исправления.

Не пытайтесь сразу сделать самый красивый тестовый каркас. Сначала добейтесь, чтобы failure был говорящим. Когда это получилось, уже проще увидеть, где нужен `maxDiff = None`, где стоит оставить короткий `msg=`, а где сообщение уже и так достаточно хорошее.

Если хотите усложнить упражнение, добавьте третий slow-тест, в котором задержка сидит не в теле теста, а в cleanup. Это хороший способ проверить на практике, что `addDuration()` учитывает cleanup-функции, а значит `--durations` покажет и такие хвосты. ([Python documentation][1])
